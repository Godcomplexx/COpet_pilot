# CoPet Pilot — общее техническое задание

## 1. Краткое описание проекта

**CoPet Pilot** — это настольный embedded-робот-компаньон на базе **ESP32-WROOM-32 DevKit**. Устройство работает как маленький интерактивный персонаж на столе: показывает лицо, меню, фокус-таймер, температуру/влажность, реагирует на движение, воспроизводит звуки и короткие анимации.

Интернет-функции не требуют постоянного телефона. ESP32 подключается к домашней сети по **Wi-Fi** и обращается по **HTTPS** к компактному CoPet Cloud API: погода, поиск, короткие AI-ответы, синхронизация времени и другие сетевые данные. Телефон или ноутбук нужен только как необязательный браузер для первоначальной настройки Wi-Fi.

В следующей аппаратной версии устройство может использовать **GPS/GNSS-модуль с антенной** для Outdoor Mode. Outdoor Mode не входит в текущий MVP.

---

## 2. Главная идея

Проект — не «чат-бот на ESP32», а физический companion-device с характером.

> **CoPet Pilot — автономный настольный companion-device с режимами, сенсорами, экраном, звуком и прямым Wi-Fi-доступом к облачным функциям.**

Разделение ответственности:

- ESP32-WROOM-32 держит всё локальное поведение: UI, режимы, датчики, анимации, звук, настройки и устойчивую работу без интернета;
- CoPet Cloud API выполняет тяжёлые сетевые операции: поиск, speech-to-text, AI и нормализацию ответов;
- телефон не является обязательной частью продукта и не участвует в обычной работе устройства.

---

## 3. Основные режимы устройства

| Режим | Назначение |
|---|---|
| **Desk Mode** | основной настольный режим |
| **Menu** | временная навигация, открываемая прокруткой энкодера |
| **Focus Mode** | режим учёбы/работы с таймером |
| **Assistant Mode** | голосовой или заранее заданный запрос через Wi-Fi/API |
| **Mini TV Mode** | воспроизведение видео/звуков с SD-карты |
| **Settings Mode** | настройки яркости, звука, Wi-Fi и темы |
| **Sleep Mode** | выключенный экран и ожидание события пробуждения |

Outdoor Mode относится к следующей версии и не отображается в меню MVP.

## 5. Роль каждого компонента

### 5.1 ESP32-WROOM-32

ESP32-WROOM-32 — главный контроллер текущей аппаратной версии.

Отвечает за:

- запуск прошивки;
- state machine режимов;
- отрисовку UI;
- чтение кнопок и энкодера;
- работу с датчиками;
- подключение к Wi-Fi;
- HTTPS-запросы к CoPet Cloud API;
- работу с SD-картой;
- воспроизведение звука;
- управление экраном;
- хранение настроек в NVS/flash.

Обычный ESP32-WROOM-32 не имеет PSRAM, поэтому архитектура должна экономить память: один компактный framebuffer, небольшие сетевые буферы, потоковая передача аудио и заранее преобразованные графические ресурсы. Полноэкранные анимации хранятся в сжатом или палитровом формате; тяжёлое декодирование и AI выполняются вне устройства.

---

### 5.2 ST7789 240×240 TFT

Экран используется как лицо и основной интерфейс CoPet Pilot.

Назначение:

- лицо робота;
- меню;
- Focus Mode timer;
- погода;
- комнатная температура/влажность;
- статусы Wi-Fi и online/offline;
- outdoor speed screen;
- Mini TV Mode;
- короткие AI-ответы.

Интерфейс: **SPI**.

Типичные пины модуля:

```text
GND
VCC
SCK
SDA / MOSI
RES
DC
BLK
```

---

### 5.3 Сенсорная кнопка

Сенсорная кнопка используется как быстрый физический control element.

Предлагаемая логика:

| Действие | Функция |
|---|---|
| короткое касание | выбрать / подтвердить |
| долгое касание | вернуться в Desk Mode |
| двойное касание | быстрый Focus Mode |
| удержание 5 сек | sleep / выключить экран |

---

### 5.4 Колесо от мыши / энкодер

Колесо используется для удобной навигации по меню.

Предлагаемая логика:

| Действие | Функция |
|---|---|
| прокрутка в Desk Mode | открыть меню и выбрать соседний пункт |
| прокрутка в Menu | предыдущий / следующий пункт |
| прокрутка в режиме | изменить параметр, если режим это поддерживает |

Текущее трёхконтактное колесо работает как rotary encoder без отдельной кнопки:

```text
A phase
B phase
COM → GND
внутренние pull-up на A/B
```

---

### 5.5 SHT31 температура/влажность

SHT31 используется для Desk Mode и функции “room comfort”.

Примеры отображения:

```text
Room: 24°C
Humidity: 48%
CoPet feels comfy
```

Поведение CoPet:

| Условие | Реакция |
|---|---|
| комфортно | happy idle |
| жарко | tired face |
| холодно | shivering animation |
| сухо | asks for water |
| влажно | uncomfortable face |

Интерфейс: **I2C**.

---

### 5.6 MPU6050

MPU6050 используется для движения, наклона и wake-up логики.

Назначение:

- определить, что устройство взяли в руки;
- определить наклон;
- детектировать тряску;
- разбудить экран по движению;
- отличать desk state от carried state;
- использовать motion events для анимаций;
- помогать Outdoor Mode.

Важно: **MPU6050 не должен быть основным источником скорости**. Скорость по акселерометру быстро накапливает ошибку. Для реальной скорости лучше использовать GPS/GNSS.

Интерфейс: **I2C**.

---

### 5.7 GPS/GNSS-модуль + антенна

GPS/GNSS используется только в следующей аппаратной версии для Outdoor Mode.

Назначение:

- скорость;
- дистанция;
- координаты;
- направление движения;
- точное время;
- количество спутников;
- last known location;
- outdoor activity log.

Схема:

```text
GPS/GNSS antenna
        ↓
GPS/GNSS receiver module
        ↓ UART
ESP32-WROOM-32
```

Важно: одна антенна без GNSS-модуля не даст данные. Нужен именно модуль-приёмник.

---

### 5.8 Динамик

Динамик используется для:

- звуков питомца;
- feedback-звуков меню;
- завершения Focus Mode;
- Mini TV audio;
- коротких voice/audio responses.

Динамик нельзя нормально подключать напрямую к GPIO ESP32. Нужен усилитель.

Рекомендуемые варианты:

| Усилитель | Комментарий |
|---|---|
| **MAX98357A** | хороший I2S mono amplifier |
| **PAM8302** | простой mono amp |
| **PAM8403** | простой stereo amp |

Предпочтительный вариант: **MAX98357A по I2S**.

---

### 5.9 Микрофон

INMP441 подключается к ESP32 по I2S и используется для коротких голосовых запросов. Распознавание речи не выполняется локально.

Поток голосового запроса:

```text
User voice
   ↓
INMP441 → ESP32 I2S RX
   ↓ потоковая отправка по HTTPS
CoPet Cloud API → speech-to-text → search/AI
   ↓ короткий JSON-ответ
ESP32 → ST7789 + MAX98357A
```

Аудио следует отправлять потоком небольшими блоками, а не сохранять целую запись в RAM. Для первого сетевого теста допустим короткий запрос без локального wake word: запись начинается только после явного действия пользователя.

---

### 5.10 SD-карта

SD-карта нужна для Mini TV Mode и хранения ресурсов.

На SD-карте можно хранить:

- короткие видео;
- картинки;
- анимации;
- звуки;
- темы;
- focus logs;
- outdoor logs;
- cached network data.

Рекомендуемый формат для видео:

```text
MJPEG 240×240, 10–15 fps
+ WAV / MP3 audio
```

Не рекомендуется пытаться проигрывать обычный YouTube/MP4/H.264 напрямую на ESP32-WROOM-32.

## 6. Общая аппаратная архитектура

```text
                       Internet
                          │ HTTPS
                          ▼
              ┌────────────────────────┐
              │    CoPet Cloud API     │
              │ weather / STT / search │
              │ AI / compact responses │
              └────────────┬───────────┘
                           │ Wi-Fi 2.4 GHz
                           ▼
┌────────────────────────────────────────────────┐
│              ESP32-WROOM-32 DevKit             │
│                                                │
│  SPI  ── ST7789 240×240 TFT                    │
│  SPI  ── SD card module, later                 │
│  I2C  ── SHT31 + MPU6050                       │
│  GPIO ── TTP223 + three-contact encoder        │
│  I2S  ── MAX98357A → speaker                   │
│  I2S  ── INMP441 microphone                    │
│  NVS  ── Wi-Fi credentials + settings          │
└────────────────────────────────────────────────┘

Phone/laptop browser (optional, setup only)
              └── Wi-Fi provisioning page
```

Устройство должно сохранять базовую функциональность без сети. Desk Mode, меню, Focus Mode, датчики и локальные анимации работают offline. Ошибка Wi-Fi не должна перезагружать устройство или блокировать UI.

## 8. Предлагаемое меню

```text
CoPet Pilot

> Focus Mode
  Assistant
  Animation Gallery
  Mini TV
  Settings
```

`Desk Mode` не является пунктом, который необходимо открывать после загрузки: это домашний экран и состояние по умолчанию.

Управление:

| Управление | Действие |
|---|---|
| колесо в Desk Mode | открыть меню |
| колесо в меню | прокрутка пунктов |
| короткое касание | выбрать пункт |
| долгое касание | вернуться в Desk Mode |
| удержание | sleep |

Если меню открыто и 10 секунд нет ввода, устройство автоматически возвращается в Desk Mode.

## 9. Desk Mode

Desk Mode — основной режим, когда устройство стоит на столе и питается от USB.

Функции:

- лицо CoPet;
- idle-анимации;
- отображение времени;
- отображение температуры/влажности;
- реакция на движение;
- sleep при отсутствии активности;
- быстрый переход в Focus Mode;
- звуки/эмоции.

Пример экрана:

```text
12:45
24°C  48%

CoPet is comfy :)
```

## 10. Focus Mode

Focus Mode — режим учёбы/работы.

Функции:

- таймер фокуса;
- пауза/продолжение;
- завершение сессии;
- короткий звук окончания;
- статистика сессий;
- геймификация питомца.

Базовые параметры:

```text
Focus: 25 min
Break: 5 min
Cycles: 4
```

Пример экрана:

```text
FOCUS
23:41 left

CoPet studies with you
```

## 11. Wi-Fi и Online Assistant

Телефонное приложение не требуется для обычной работы. ESP32-WROOM-32 подключается к Wi-Fi в station mode и выполняет HTTPS-запросы.

### 11.1 Почему не отправлять запрос прямо в поисковую страницу

HTML-страница поисковика не является стабильным API: она тяжёлая, меняет разметку и неудобна для разбора на микроконтроллере. Для устройства нужен структурированный endpoint, возвращающий небольшой JSON.

Предпочтительный поток:

```text
encoder/touch or short microphone recording
        ↓
assistant_service on ESP32
        ↓ HTTPS POST
CoPet Cloud API
        ├── weather provider
        ├── speech-to-text provider
        ├── search provider
        └── AI provider
        ↓ compact JSON, normally <= 2 KB
ESP32 displays a card and optional local sound
```

CoPet Cloud API — небольшой backend-адаптер, а не микросервисная система. Он нужен для трёх задач:

1. не хранить ключи сторонних API внутри прошивки;
2. ограничивать размер и время ответа;
3. возвращать единый простой формат независимо от внешнего поставщика.

Минимальный device-facing контракт описан в
`docs/architecture/cloud_api_contract.md`.

Для первого прототипа разрешён прямой HTTPS-запрос к одному публичному API без секретного ключа. Ключи платных сервисов в прошивку и Git добавлять нельзя.

### 11.2 Сетевые функции MVP

| Функция | Где выполняется |
|---|---|
| Wi-Fi reconnect | ESP32 |
| время через SNTP | ESP32 |
| запрос погоды | ESP32 → API |
| запись короткого аудио | ESP32 |
| speech-to-text | CoPet Cloud / внешний provider |
| поиск и AI-ответ | CoPet Cloud / внешний provider |
| отображение и локальная реакция | ESP32 |

Пример запроса:

```json
{
  "request_id": "42",
  "type": "assistant_query",
  "text": "Какая погода сегодня?",
  "locale": "ru-RU"
}
```

Пример ответа:

```json
{
  "request_id": "42",
  "type": "assistant_answer",
  "text": "Сегодня около 18 градусов, возможен дождь.",
  "mood": "helpful",
  "ttl_sec": 300
}
```

### 11.3 Настройка Wi-Fi

Если учётные данные ещё не сохранены, Desk Mode всё равно запускается. В Settings пользователь выбирает `Wi-Fi Setup`; ESP32 временно создаёт точку доступа `CoPet-Setup` и локальную страницу для ввода SSID/пароля. Страницу можно открыть с телефона или ноутбука. После успешного подключения данные сохраняются в NVS.

Mini TV Mode — режим проигрывания локальных коротких видео/анимаций с SD-карты.

Рекомендуемый подход:

```text
SD card
  ↓
MJPEG frames
  ↓

```

Для звука:

```text
SD card audio
  ↓
ESP32-WROOM-32 I2S
```

- стримить YouTube напрямую на ESP32;
- передавать видео по BLE;
- начинать с MP4/H.264;

Рекомендуемый формат:

```text
/video/focus_intro.mjpeg
/video/rain_loop.mjpeg
/video/copet_idle.mjpeg
/audio/focus_intro.wav
/audio/rain_loop.wav
```

## 13. Outdoor Mode

**Статус: отложено до следующей версии.** Outdoor Mode не входит в MVP, не отображается в текущем меню и не должен влиять на архитектуру первой версии. Раздел сохраняется как направление развития.

В будущей версии Outdoor Mode используется, когда устройство снято со стола и закреплено на рюкзак/сумку/велосипед.

Функции:

- скорость;
- дистанция;
- состояние движения;
- ходьба/езда;
- направление движения;
- last known location;
- outdoor animation.

Источники данных:

| Данные | Источник |
|---|---|
| скорость | GPS/GNSS |
| координаты | GPS/GNSS |
| дистанция | GPS/GNSS |
| движение/тряска | MPU6050 |
| подняли/положили | MPU6050 |
| outdoor state | GPS + MPU6050 |

Пример экрана:

```text
Outdoor Pilot
Speed: 12.6 km/h
Distance: 1.4 km
```

## 14. Software architecture

Прошивка строится как модульный монолит. Отдельные FreeRTOS-задачи используются только для реально параллельных потоков: UI/app loop, audio, network и при необходимости media. Модули общаются через типизированные события и небольшие очереди; драйверы не меняют режим приложения напрямую.

Целевая структура проекта:

```text
main/
  app_main.c

  core/
    app_state.c/.h        current mode and transitions
    app_events.c/.h       typed input/sensor/network events
    app_config.c/.h       build-time pin and feature configuration

  modes/
    desk_mode.c/.h
    menu_mode.c/.h
    focus_mode.c/.h
    assistant_mode.c/.h
    animation_mode.c/.h
    mini_tv_mode.c/.h
    settings_mode.c/.h

  ui/
    display.c/.h
    primitives.c/.h
    face_ui.c/.h
    menu_ui.c/.h
    cards_ui.c/.h
    focus_ui.c/.h

  drivers/
    st7789.c/.h
    touch_button.c/.h
    rotary_encoder.c/.h
    sht31.c/.h
    mpu6050.c/.h
    audio_i2s.c/.h
    sdcard.c/.h          later MVP stage

  services/
    sensor_service.c/.h
    focus_service.c/.h
    animation_service.c/.h
    wifi_service.c/.h
    http_service.c/.h
    assistant_service.c/.h
    time_service.c/.h
    settings_service.c/.h
    media_service.c/.h   later MVP stage

  assets/
    animations/
    sounds/
```

Ответственность слоёв:

| Слой | Отвечает за | Не должен |
|---|---|---|
| drivers | регистры, GPIO, SPI/I2C/I2S | выбирать экран или режим |
| services | датчики, Wi-Fi, HTTP, таймеры, медиа | рисовать UI напрямую |
| modes | правила конкретного режима | обращаться к регистрам железа |
| ui | построение framebuffer | выполнять сеть и бизнес-логику |
| core | события и переходы состояний | содержать код конкретного драйвера |

`app_main.c` только инициализирует модули, создаёт задачи и запускает главный цикл. Он не должен содержать реализацию дисплея, датчиков, сети или экранов.

### 14.1 Runtime tasks

```text
app/ui task
  consumes input + service events
  updates state machine
  renders dirty screen

network task
  Wi-Fi reconnect + one HTTPS request at a time
  posts NETWORK_* / ASSISTANT_* events

audio task
  owns I2S RX/TX
  streams bounded audio blocks

sensor polling
  periodic timer or low-priority task
  posts validated samples
```

На WROOM одновременно выполняется только один тяжёлый сетевой запрос. UI не ждёт HTTP синхронно.

Иллюстрированные user cases, wireframe экранов и карта переходов находятся в

Desk Mode является домашним состоянием. После инициализации устройство сразу показывает лицо; меню появляется только при прокрутке колеса.

Основные состояния приложения:

```text
BOOT
  └── hardware init complete ──> DESK_MODE

DESK_MODE
  ├── encoder rotate ──────────> MENU
  ├── double touch ────────────> FOCUS_MODE
  └── hold 5 sec ──────────────> SLEEP

MENU
  ├── encoder rotate ──────────> change selection
  ├── short touch ─────────────> selected mode
  ├── long touch ──────────────> DESK_MODE
  └── 10 sec timeout ──────────> DESK_MODE

FOCUS_MODE / ASSISTANT_MODE / ANIMATION_MODE /
MINI_TV_MODE / SETTINGS_MODE
  └── long touch ──────────────> DESK_MODE

SLEEP
  └── touch or motion ─────────> DESK_MODE
```

Состояние сети идёт параллельно состоянию UI и не создаёт отдельные комбинации режимов:

```text
NET_UNCONFIGURED
NET_CONNECTING
NET_ONLINE
NET_OFFLINE
NET_ERROR
```

Assistant Mode имеет собственные под-состояния:

```text
ASSISTANT_IDLE
ASSISTANT_RECORDING
ASSISTANT_UPLOADING
ASSISTANT_WAITING
ASSISTANT_RESULT
ASSISTANT_ERROR
```

При `NET_OFFLINE` устройство остаётся в текущем локальном режиме и показывает небольшой offline-индикатор. Переход в `NET_ERROR` не является причиной reboot.

## 16. MVP

MVP должен доказать, что устройство работает как самостоятельный embedded-продукт.

### MVP hardware

```text
ESP32-WROOM-32 DevKit
ST7789 TFT
three-contact mouse wheel
TTP223 touch button
SHT31
MPU6050
MAX98357A + speaker
INMP441 microphone
USB power
```

SD-карта добавляется после стабильной локальной архитектуры и Wi-Fi. GPS не входит в MVP.

### MVP software

```text
- загрузка устройства
- запуск сразу в Desk Mode
- меню, открываемое энкодером
- Focus Mode
- чтение SHT31
- чтение MPU6050
- базовые анимации лица
- звуки через динамик
- Wi-Fi setup/reconnect
- время и один простой HTTPS-запрос
- короткая карточка ответа от CoPet Cloud API
```

Голосовой запрос, SD/Mini TV и Outdoor развиваются отдельными этапами после базового MVP. Существующий BLE-тест может остаться диагностическим модулем, но Phone Bridge больше не является обязательной продуктовой функцией.

## 17. Ревизии разработки

### Rev 1 — Desk Companion

Цель: собрать настольного робота.

Функции:

- экран;
- меню;
- кнопка;
- энкодер;
- Desk Mode;
- Focus Mode;
- температура/влажность;
- motion reactions;
- базовый звук.

---

### Rev 2 — Firmware Architecture

Цель: разделить работающий аппаратный тест на устойчивые модули без изменения поведения.

Функции:

- `app_state` и типизированные события;
- отдельные display/input/sensor/audio drivers;
- Desk-first state machine;
- settings в NVS;
- MPU6050 после получения модуля;
- host-тесты чистой логики таймера и переходов.

---

### Rev 3 — Direct Wi-Fi

Цель: дать устройству интернет без обязательного телефона.

Функции:

- Wi-Fi provisioning через SoftAP;
- сохранение credentials в NVS;
- reconnect без блокировки UI;
- SNTP time;
- HTTPS client;
- одна online-карточка, например погода.

---

### Rev 4 — Online Assistant

Цель: отправлять короткий запрос в CoPet Cloud API и показывать компактный ответ.

Функции:

- заранее заданные запросы из меню;
- короткая запись с INMP441;
- потоковая отправка аудио;
- cloud speech-to-text;
- search/AI response;
- экран `recording / waiting / result / error`.

---

### Rev 5 — Mini TV

Цель: добавить локальное медиа после стабилизации RAM и SPI.

- SD-карта на отдельной SPI-шине или после решения CS-конфликта дисплея;
- список ресурсов;
- палитровые изображения или MJPEG с низкой частотой кадров;
- WAV audio;
- bounded buffers без полного файла в RAM.

---

### Rev 6 — Outdoor Pilot, следующая версия

Цель: добавить GPS/GNSS и outdoor-функции после завершения настольной версии.

---

## 18. Критерии готовности

### Устройство считается рабочим MVP, если:

- включается и сразу показывает Desk Mode;
- прокрутка энкодера открывает меню;
- энкодером можно прокручивать пункты;
- кнопкой можно выбирать режимы;
- Desk Mode показывает лицо/статус;
- SHT31 отдаёт температуру и влажность;
- MPU6050 реагирует на движение/наклон;
- Focus Mode запускает таймер;
- динамик воспроизводит хотя бы короткий звук;
- настройки Wi-Fi сохраняются;
- offline-режим не блокирует локальные функции;
- HTTPS-запрос возвращает одну короткую карточку.

### Расширенная версия считается готовой, если:

- CoPet получает погоду напрямую через Wi-Fi;
- голосовой запрос проходит через INMP441 и CoPet Cloud API;
- SD-карта и Mini TV работают без зависания UI;
- GPS показывает скорость;
- Outdoor Mode считает дистанцию;
- данные логируются;
- корпус позволяет использовать устройство на столе и на рюкзаке.

## 19. Что не делать в начале

Не включать в MVP:

- полноценный YouTube-плеер;
- потоковое видео по BLE;
- локальный LLM на ESP32;
- полноценное распознавание речи на ESP32;
- хранение ключей сторонних API в прошивке или Git;
- парсинг HTML-страниц поисковиков на ESP32;
- обязательное телефонное приложение до появления доказанной необходимости;
- сложный аккумуляторный power-management;
- слишком много режимов сразу;
- большой тачскрин;
- камеру.

## 20. Структура прошивки в репозитории

ESP-IDF project root лежит в корне репозитория, активные исходники — в `main/`.

```text
main/
  app_main.c
  core/       state machine, events, config
  modes/      desk, menu, focus, assistant, animation, mini_tv, settings
  ui/         display, primitives, face, menu, cards, focus
  drivers/    ST7789, touch, encoder, SHT31, MPU6050, I2S, SD
  services/   sensors, focus, animation, Wi-Fi, HTTP, assistant, time, settings
```

Текущий репозиторий уже содержит работающий аппаратный тест, но `main/app_main.c` пока объединяет несколько слоёв. Следующая архитектурная задача — переносить работающий код в указанную структуру небольшими проверяемыми шагами, сохраняя поведение после каждого шага.
