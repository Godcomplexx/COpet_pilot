# ADR-002: Offline-first и прямой Wi-Fi через CoPet Cloud API

## Статус

Принято — 2026-07-16.

## Контекст

Первоначальная схема делала телефон обязательным BLE-мостом для погоды,
голоса и AI. ESP32-WROOM-32 уже имеет Wi-Fi и ESP-IDF HTTP/S client. При этом
прямой вызов множества сторонних API из прошивки раскрывает ключи, усложняет
TLS и привязывает устройство к форматам разных поставщиков.

## Рассмотренные варианты

| Вариант | Плюсы | Минусы |
|---|---|---|
| телефон как BLE-мост | ключи остаются в приложении | обязательное приложение и соединение |
| прямые provider API | нет собственного backend | ключи в прошивке, разные большие ответы |
| один CoPet Cloud gateway | единый короткий контракт, ключи вне устройства | нужен маленький backend |

## Решение

Локальные функции работают offline. Для online-функций ESP32 подключается по
Wi-Fi и HTTPS к одному небольшому CoPet Cloud API. Телефон/ноутбук используется
только как необязательный браузер настройки Wi-Fi.

## Компромиссы и последствия

- появляется backend, который нужно развернуть и мониторить;
- без сети online-функции недоступны, но локальные режимы продолжают работать;
- ответы ограничиваются компактным JSON, обычно не более 2 КБ;
- сторонние API keys хранятся только на backend;
- на устройстве одновременно выполняется один тяжёлый HTTPS-запрос.

## Пересмотреть решение, если

- появится функция, действительно требующая постоянного мобильного приложения;
- выбранный внешний API безопасно и стабильно работает напрямую без секрета;
- backend становится сложнее самого продукта.
